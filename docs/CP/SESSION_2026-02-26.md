## What Was Done

### 1. Root cause investigation — "OTP service temporarily unavailable"

Registration screen on demo was returning this error. Three layered bugs found:

| # | Bug | Location | Fix | PR |
|---|-----|----------|-----|----|
| 1 | Migrations 016-020 never applied → Plant returned 500 | Demo DB | Applied SQL manually via PP portal ✅ | — |
| 2 | CP Backend mapped all Plant 5xx → bare 502 on registration OTP path | `cp_registration_otp.py` | `>=500` → 503 UPSTREAM_ERROR + Retry-After | #805 ✅ merged |
| 3 | Plant Gateway `BudgetGuardMiddleware` returned 500 for `/api/v1/otp/sessions` because `jwt_claims=None` on CP-Registration-Key paths | `budget.py` | Added otp/sessions to skip list | #806 ✅ merged |
| 4 | `send_otp_email.delay()` is a synchronous Celery call — blocked until Redis accepts message; CP Backend's 10s httpx timeout fired, session committed to DB but 201 never returned | `Plant/BackEnd/api/v1/otp.py` | Skip email in non-prod; use `BackgroundTasks` in prod | #807 ⏳ pending deploy |

**Key evidence that led to bug #4**: OTP session row existed in DB (`attempts=0`, `verified_at=null`) but CP got a network exception, not a 5xx. Means Plant committed the INSERT then hung before responding — Celery `delay()` blocking on Redis (not running in demo).

CP Frontend

          Middleware stack: Auth → RBAC → Policy → Budget → ErrorHandling → Proxy
