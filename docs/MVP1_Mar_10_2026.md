# MVP1 — Frontend Journey Completion Plan

**Plan ID:** `MVP1`  
**Created:** 2026-03-10  
**Author:** GitHub Copilot  
**Branch context:** `feat/ui-ux-revamp` / PR #917 merged to `main`; release-readiness follow-ups committed on branch  
**Primary surfaces:** CP FrontEnd, PP FrontEnd, mobile, with thin-proxy/backend support only where existing contracts are insufficient  
**Primary source of truth:** `docs/CONTEXT_AND_INDEX.md`  

## Vision Intake

- **Service area:** CP FrontEnd, PP FrontEnd, mobile, plus only the smallest required CP/PP/Plant contract changes to complete the user journey.
- **User outcome:** A customer can discover an agent, start or resume the hire flow, operate the hired runtime from web or mobile, and understand what operators did in PP without losing context.
- **Out of scope:** New marketplace categories, pricing model changes, net-new design-system rewrite, deep infrastructure refactors, non-MVP backend platform redesign.
- **Lane choice:** Start with Lane A wherever an existing API already exists; use Lane B only when a missing proxy or public route blocks truthful UI behavior.
- **Timeline constraint:** Ship as five mergeable iterations, each capable of standing on its own without waiting for all five to finish.

## MVP Outcome

When all five iterations are complete, WAOOAW should behave like one coherent product instead of three partially connected surfaces. The MVP is complete when the main customer journey, the operator intervention loop, and the regression gates are all truthful, continuous, and reviewable.

## Iteration Summary

| Iteration | Theme | Outcome | Stories | Est. effort |
|---|---|---|---:|---:|
| 1 | Journey completion | Customer and operator can complete the primary journey without dead ends or context loss | 3 | 1 week |
| 2 | Live data convergence | Key CP, PP, and mobile runtime surfaces show backend-truthful loading, empty, error, and success states | 3 | 1 week |
| 3 | Operator loop MVP | PP actions become visible and meaningful to customers in CP and mobile | 4 | 1 to 1.5 weeks |
| 4 | E2E reliability | The main journeys are protected by stable browser and mobile regression coverage | 4 | 1 week |
| 5 | Release readiness | Warnings, smoke gates, and documentation are cleaned up for release-quality handoff | 4 | 1 week |

## Execution Rules For Future Agents

1. Execute one story at a time. Do not combine stories across iterations in a single PR unless the story card explicitly says so.
2. Read only the files listed in the story card's `Files to read first` block before editing.
3. For CP or PP API work, keep thin-proxy discipline. Do not add business logic to CP BackEnd or PP BackEnd.
4. Any new FastAPI route must use `waooaw_router()` and app-level dependencies already established by the service.
5. Any new GET route must use the read session dependency, not the write session dependency.
6. Prefer existing API contracts and existing hooks first. Do not invent a new endpoint if a route already exists and only the response shape or mapping is wrong.
7. Every story must end with the exact validation command listed in its card.
8. After completing a story, update the story status in this file from `Planned` to `Done` in the story header before opening the PR.

## Inline Backend Patterns

Use these exact patterns when a story requires a new or extended backend route.

### Pattern A — Standard router

```python
from fastapi import Depends
from core.routing import waooaw_router

router = waooaw_router(prefix="/resource", tags=["resource"])
```

### Pattern B — Read-only database dependency

```python
from fastapi import Depends
from sqlalchemy.ext.asyncio import AsyncSession
from core.database import get_read_db_session

@router.get("/items")
async def list_items(
    db: AsyncSession = Depends(get_read_db_session),
):
    ...
```

### Pattern C — CP BackEnd thin proxy rule

Use an existing `api/cp_*.py` thin-proxy pattern when the CP FrontEnd needs a new shape. Keep the route as pass-through or light response mapping only. Do not place business logic, storage, or policy enforcement in CP BackEnd.
